# RoboMaster 步兵电控技能

## 固定硬件背景

默认按以下硬件背景理解需求，除非用户明确说明有变更：

- 车型：RoboMaster 步兵
- 主控：STM32F407
- 控制对象：自瞄云台、摩擦轮、拨弹机构
- 云台电机：
  - yaw 电机：DN4310，减速比 1:10
  - pitch 电机：DN4310，减速比 1:10
- 发射机构电机：
  - 摩擦轮电机：M3508
  - 拨弹电机：M2006，减速比 1:36
- IMU：HI12H3
- IMU 频率：250 Hz

如果用户后续给出新的型号、减速比、频率或任务周期，以用户提供的信息覆盖默认背景。

## 强制执行约束

### 1. 修改代码前先征求用户确认

在任何代码改动前，必须先完成以下步骤：

1. 说明要改哪些文件、为什么改、预期效果是什么。
2. 给出拟修改代码或 diff 片段，供用户查验。
3. 明确等待用户确认。
4. 只有在用户确认后，才能实际修改文件。

如果用户只是在讨论方案、调参思路或架构选择，则只给方案，不直接落盘修改。

### 2. 保护中文注释和文件编码

处理代码时必须遵守以下规则：

- 不改变原文件编码，避免把原中文注释改成乱码。
- 不因格式化、整文件重写、批量替换而影响未修改区域的注释。
- 新增注释必须使用中文。
- 如果发现文件本身已有乱码注释，默认不主动“顺手修复”，除非用户明确要求处理乱码。
- 修改时尽量最小化 patch，只改与当前任务直接相关的代码行。

## 输出优先级

优先输出以下高价值内容：

- 控制链路和任务周期是否合理
- 角度环、速度环、电流/力矩控制链是否闭环一致
- yaw/pitch 方向定义、减速比、反馈量纲是否统一
- 摩擦轮与拨弹机构联动逻辑是否完整
- 卡弹检测、停火保护、热量限制是否闭环
- IMU 250 Hz 数据在控制周期中的使用是否合理
- CAN/UART 通信是否可能成为延迟或失步瓶颈
- 可以量化验证的调参指标和测试步骤

## 工作流程

### 1. 先明确上下文

给出建议前，优先确认这些信息：

- 当前目标：修 bug、重构、调参、加功能、赛前排障
- 涉及模块：云台、视觉、自瞄、摩擦轮、拨弹、遥控、通信
- 控制周期：任务频率、中断频率、IMU 更新频率、控制刷新频率
- 反馈来源：IMU、编码器、电机回传、视觉数据
- 当前症状：抖动、超调、反向、延迟、丢包、卡弹、发热、断联
- 验证手段：串口日志、上位机曲线、示波器、裁判系统信息

如果信息不足，先列出假设，再基于保守假设给方案。

### 2. 分模块分析

默认按以下模块拆解：

- 云台：
  - yaw/pitch 电机选型、减速比、方向、零点、限位
  - 角度环、速度环、前馈、滤波、视觉融合
- 摩擦轮：
  - 启停时序、目标转速、左右轮一致性、热量与供电限制
- 拨弹机构：
  - 单发/连发逻辑、角度换算、卡弹检测、反卡弹策略
- 感知与估计：
  - HI12H3 250 Hz 数据使用方式、姿态稳定性、时间戳一致性
- 通信与任务：
  - CAN/UART 帧设计、任务优先级、阻塞风险、状态同步

### 3. 代码修改前的答复格式

如果用户要求改代码，先按这个格式回答：

1. 假设与约束
2. 拟修改文件
3. 修改思路
4. 拟修改代码或 diff
5. 验证方法
6. 等待用户确认

未获得确认前，不执行实际文件修改。

### 4. 调参与排障

调参和排障时，优先使用“现象 -> 根因假设 -> 验证方法 -> 修改建议 -> 风险”的结构。

重点关注以下典型问题：

- 云台抖动：
  - 反馈噪声过大
  - 环路频率不匹配
  - 减速比或方向定义错误
  - 视觉量和 IMU 量混用
- pitch/yaw 超调：
  - 角速度反馈不足
  - 微分或阻尼不够
  - 前馈不合理
- 摩擦轮转速不稳：
  - 给定斜坡缺失
  - 左右轮参数不一致
  - 供电波动或限流影响
- 拨弹卡弹：
  - 单发角度换算不准
  - 力矩阈值不合理
  - 卡弹处理状态机不完整
- 视觉自瞄效果差：
  - 坐标系不统一
  - 视觉延迟未补偿
  - 目标角速度前馈缺失

### 5. 默认输出要求

默认输出应包含：

1. 当前假设
2. 问题定位
3. 拟修改方案
4. 修改前需用户确认的代码片段
5. 验证步骤
6. 风险和回退方案
